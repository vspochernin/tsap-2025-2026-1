# Введение

В рамках данного проекта вам предстоит написать автоматизированные тесты для веб- и мобильного приложения. Задания имитируют реальную задачу: необходимо проверить корректность работы пользовательских интерфейсов сайта и Android-приложения. Для этого используйте знания из пройденных модулей: Java, Selenium, TestNG для веб-тестирования и Appium для мобильного тестирования. Результатом будет готовый проект с тестовыми сценариями, который можно добавить в портфолио.

## Что нужно сделать

### Часть 1. Автотесты сайта

Выберите **публичный сайт** с простым пользовательским интерфейсом (например, информационный портал, онлайн-каталог или новостной сайт без авторизации и сложной бизнес-логики). Изучите основные сценарии работы сайта (навигация по разделам, поиск, работа форм и так далее) и реализуйте для них автоматизированные тесты на Selenium. Покройте хотя бы четыре ключевых сценария (но лучше больше). Например, проверьте, что при клике на пункт меню открывается правильная страница, форма поиска возвращает результаты, элементы на странице отображаются корректно.

### Часть 2. Автотесты Android-приложения Wikipedia

Напишите автоматизированные тесты для **Android-приложения Wikipedia** с помощью Appium. Задачи могут включать проверку отображения главного экрана, ввод текста в строку поиска, открытие статьи и проверку её заголовка, переключение языковых настроек или прокрутку страницы. Убедитесь, что тесты выполняются на эмуляторе Android и стабильно проверяют заданные сценарии.

#### Обязательные требования к покрытию:

- Реализуйте не менее двух стабильных сценариев.
- Для получения максимального балла за покрытие рекомендуется реализовать три и более сценария.

#### Примеры сценариев:

- Поиск статьи по ключевому слову и проверка того, что открылась корректная страница.
- Проверка заголовка открытой статьи.
- Прокрутка статьи до определённого элемента и проверка его наличия.
- Смена языка и проверка, что поиск/контент работают на выбранном языке.

Тесты должны стабильно выполняться на эмуляторе Android (или реальном устройстве) и проверять ожидаемый результат.

## Технические требования

### Стек технологий:

- Язык: Java (рекомендуется версия 11 или выше).
- Сборка: Maven или Gradle.

### Фреймворки:

- Для веб-тестов: Selenium WebDriver, TestNG;
- Для мобильных тестов: Appium Java Client, TestNG.

Рекомендуется использовать WebDriverManager для автоматической загрузки драйверов браузеров.

### Мобильное тестирование:

- Используйте Appium Server (локальный или через Appium Desktop).
- Тесты должны выполняться на Android-эмуляторе или реальном устройстве.
- Предварительно установите APK Wikipedia (с официального источника).

### Структура и минимальные требования:

- Проект организован как тестовый проект, с разумным разделением на пакеты (например: pages, tests, utils и тому подобное).
- Для веб-тестов рекомендуется использовать Page Object Model (как минимум для ключевых страниц/экранов).
- Допускается базовый уровень архитектуры (сложные паттерны не требуются), но код должен быть читаемым и поддерживаемым.

### Дополнительно (по желанию):

- Можно добавить несколько простых API-тестов (например, проверка ответа открытого API), однако основной фокус - UI-тестирование.
- Наличие API-тестов не обязательно и не добавляет отдельные баллы, но может положительно повлиять на итоговую оценку.

## Этапы выполнения проекта

- **Подготовка окружения**. Убедитесь, что у вас установлен JDK (11+), настройте Maven/Gradle. Добавьте зависимости Selenium, TestNG, Appium и другие. Организуйте структуру проекта (например, src/test/java).
- **Исследование сайта**. Выберите сайт и изучите его структуру: определите элементы интерфейса, которые будете тестировать (кнопки, ссылки, поля ввода). Сформулируйте тестовые сценарии (например: открыть главную страницу, перейти в раздел, выполнить поиск, проверить корректность результатов).
- **Разработка веб-тестов**. Создайте классы Page Object для основных страниц сайта (если используете этот подход). Напишите тестовые методы (TestNG), реализующие выбранные сценарии. Используйте методы WebDriver (поиск элементов, клики, ввод текста, ожидания). Добавляйте понятные проверки (assert) для ключевых результатов (например, проверка заголовка страницы или наличия текста).
- **Настройка Appium**. Установите и запустите Appium Server, настройте Android-эмулятор (или подключите устройство), убедитесь, что приложение Wikipedia доступно для тестирования (APK установлен на устройство). Настройте capabilities (например, appPackage, appActivity или путь к APK).
- **Разработка мобильных тестов**. Определите сценарии для Wikipedia: например, открыть приложение, дождаться загрузки главного экрана, найти элемент строки поиска, выполнить поиск статьи по ключевому слову, открыть результат и проверить заголовок. Используйте селекторы Appium (id, accessibilityId, xpath и так далее) и TestNG для организации тестов.
- **Запуск и отладка**. Запустите тесты (через TestNG XML или IDE), исправьте ошибки локаторов или синхронизации. Добейтесь, чтобы тесты стабильно выполнялись без неявных задержек и случайных падений.
- **Документирование**. Напишите в README.md инструкции по запуску проекта: какие зависимости нужны, как запустить тесты (например, mvn test или через TestNG XML), какие параметры можно передать. Укажите, на каких браузерах и устройствах проверено, приведите пример отчётов.

## Структура проекта и рекомендации по реализации

- **Используйте Page Object Model для веб-тестов**: создайте класс для каждой страницы сайта, описывающий локаторы и методы действий. Это облегчит поддержку тестов при изменении интерфейса. Для мобильных тестов можно аналогично создать классы-страницы (Page Object) или хранить локаторы в отдельных классах.
- **Организуйте пакеты по функциям**: например, tests.web и tests.mobile или pages.web и pages.mobile.
- **Конфигурируйте параметры (URL сайта, путь к APK, настройки драйвера)** через файл src/test/resources/config.properties или прямо в коде (главное - чтобы было понятно).
- **Используйте TestNG-аннотации (@BeforeClass, @AfterClass, @Test и так далее)** для инициализации WebDriver/AppiumDriver, открытия/закрытия приложений.
- **По возможности добавьте DataProvider** для параметризации тестов (например, несколько наборов данных для поиска).
- **Обрабатывайте ожидания (WebDriverWait)** для элементов, которые появляются динамически, чтобы тесты не падали из-за несвоевременной загрузки.
- **Следите за качеством кода**: понятные имена методов и переменных, минимизация дублирования, комментарии к нетривиальным моментам.

## Критерии оценивания

Максимальная суммарная оценка за проект - 50 баллов. Оценивание разделено по категориям, соответствующим реализации ключевых частей задания. Ниже перечислены критерии с указанием достижений и баллов.

### 1. Автотесты для сайта - максимум 20 баллов

**Покрытие тестами основных сценариев - 10 баллов**.

- 10 баллов - реализовано четыре стабильных ключевых сценария или более (поиск, навигация, переходы, формы, отображение элементов). Тесты надёжные, повторяемые.
- 7 баллов - реализовано три стабильных сценария, есть небольшие недочёты.
- 4 балла - реализовано один-два сценария, тесты нестабильны.
- 0 баллов - веб-тестов нет или они не выполняются.

**Качество реализации тестов и техника автоматизации - 6 баллов**.

- 6 баллов - используются ожидания, аккуратно реализованы Page-Object-классы, локаторы выделены корректно, логика взаимодействия с элементами вынесена из тестов.
- 3 балла - частично используется Page Object, есть дублирование, но структура в целом прослеживается.
- 0 баллов - код хаотичен, локаторы и логика вперемешку, поддержка тестов затруднена.

**Настройка окружения и запуск тестов - 4 балла**.

- 4 балла - полностью корректный запуск, README содержит инструкции по веб-тестам.
- 2 балла - работает, но требуется ручная донастройка.
- 0 баллов - запуск веб-тестов невозможен.

### 2. Автотесты для мобильного приложения Wikipedia - максимум 20 баллов

**Покрытие мобильных сценариев - 10 баллов**.

- 10 баллов - реализовано три стабильных сценария или более (поиск статьи, открытие статьи, проверка заголовка, прокрутка, язык).
- 7 баллов - выполнено два сценария, но не полностью стабильно.
- 4 балла - выполнен один сценарий, есть частые падения.
- 0 баллов - тестов нет или не запускаются.

**Корректность настройки Appium и окружения - 6 баллов**.

- 6 баллов - настроены capabilities, запуск на эмуляторе/устройстве воспроизводим по README.
- 3 балла - запуск возможен, но с ручными доработками.

**Качество кода мобильных тестов - 4 балла**.

- 4 балла - логичная структура, правильная работа с локаторами, минимальные дубли.
- 2 балла - структура частично есть.
- 0 баллов - код нечитабелен или явно некорректен.

### 3. Качество проекта, структура, архитектура - максимум 5 баллов

- 5 баллов - логичное разделение пакетов, базовый Page Object, единый стиль.
